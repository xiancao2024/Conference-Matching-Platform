\section{Objectives and Introduction}

Large conferences create two linked discovery problems: participants need help finding relevant sessions, and they also need help identifying other attendees with overlapping interests. This project builds a conference matching platform that works directly on real attendance logs instead of on curated profile cards.

The current system uses the public Kaggle Event Attendance Dataset (\texttt{cankatsrc/event-attendance-dataset}). Raw CSV rows are normalized into a conference schema with attendee entities and event/session entities. On top of that normalized data, the platform provides a hybrid retrieval engine, a local web interface, weak-label evaluation, and an optional local LLM layer for conversational responses and lightweight result explanations.

The main goal is practical retrieval: accept natural-language requests such as ``find climate attendees'' or ``show AI sessions'', map them into a structured query, and rank the most relevant imported entities using lexical, conceptual, structural, and optional embedding-based evidence.

\section{Background and Related Work}

\textbf{Dataset context.} The Kaggle dataset contains noisy operational records rather than curated user profiles. Typical fields include event identifiers, event names, locations, timestamps, attendee names, emails, and phone numbers. Useful matchmaking signals therefore have to be inferred from sparse event metadata.

\textbf{Related work.} The project is inspired by hybrid retrieval systems used in search and RAG pipelines. Instead of retrieving document passages, this system retrieves structured entities representing attendees and event resources. The implementation combines sparse lexical matching, concept expansion through a small ontology, structured boosts, and optional dense retrieval with Sentence Transformers plus FAISS.

\section{Approach and Implementation}

\subsection{Data ingestion and normalization}

The import pipeline accepts a Kaggle zip file, a CSV file, or an extracted directory. It searches for the best matching CSV using column alias rules, reads attendance rows, and writes a normalized JSON dataset.

Normalization produces two entity types:

\begin{itemize}
\item \textbf{Attendees}: role \texttt{participant}, with name, inferred organization, sector tags, asks/offers, and the list of source events they attended.
\item \textbf{Events}: stored as \texttt{resource} entities with role \texttt{session}, with name, location, date metadata, attendee counts, and inferred topical sectors.
\end{itemize}

Several fields are derived heuristically: organization from email domains, sector tags from event metadata, and short bios from attendance history. This design keeps the pipeline robust even though the source dataset does not contain explicit role labels or rich biographies.

\subsection{Hybrid retrieval engine}

The matcher in \texttt{conference\_matching/engine.py} scores entities with several signals:

\begin{itemize}
\item lexical similarity from TF-IDF-style cosine similarity over normalized tokens
\item concept similarity from ontology-driven phrase and token mappings
\item structured boosts for role fit, sector fit, stage fit, ask/offer overlap, and intent fit
\item optional dense similarity from \texttt{all-MiniLM-L6-v2} when local embeddings and FAISS are available
\end{itemize}

The final score is a weighted combination of lexical, concept, embedding, and structured components. For topical people-search queries, the matcher also applies stricter topic and event-link filtering through the attendee \texttt{source\_events} field.

\subsection{Query routing and web application}

\texttt{conference\_matching/server.py} provides a lightweight HTTP server and routes requests into four modes: empty query, greeting/thanks, general question, and retrieval request.

General questions are answered through the optional local Ollama integration in \texttt{conference\_matching/llm.py}. Retrieval requests are converted into a structured query profile and passed to the matcher. The browser client renders ranked session and attendee cards with short summaries.

The deterministic engine already produces rule-based explanations. If Ollama is available, the system also adds an extra one-sentence \texttt{llm\_reason} to top retrieved items, but retrieval itself does not depend on the LLM.

\subsection{Code organization}

\begin{itemize}
\item \texttt{conference\_matching/data.py}: dataset discovery, CSV loading, normalization, and dataset access
\item \texttt{conference\_matching/kaggle\_import.py}: CLI entrypoint for import
\item \texttt{conference\_matching/ontology.py}: phrase/token concept mappings and role aliases
\item \texttt{conference\_matching/engine.py}: indexing, scoring, ranking, explanations, and keyword baseline
\item \texttt{conference\_matching/llm.py}: optional Ollama-based general answers and top-result explanations
\item \texttt{conference\_matching/evaluation.py}: weak-label benchmark construction and ranking metrics
\item \texttt{conference\_matching/server.py}: local API and static file server
\end{itemize}

\section{Data and Analysis}

\textbf{Data source}: \url{https://www.kaggle.com/datasets/cankatsrc/event-attendance-dataset}

The source data is intentionally limited. It tells us who attended which event, but not why they attended, what role they play professionally, or what they are explicitly looking for. Because of that, the project leans on conservative inference rather than aggressive enrichment.

The main analytical choices are to treat attendance as the strongest observed relation, infer topical sectors from event metadata, and keep attendee and session entities in one searchable space.

\section{Results and Evaluation}

\subsection{Evaluation methodology}

The evaluation module builds weak labels directly from attendance history. Each attendee becomes a query, and the sessions/events they are known to have attended become the relevant set. The system compares the full hybrid matcher against a keyword-only baseline using Precision@5, Recall@5, nDCG@5, and Mean Reciprocal Rank (MRR).

\subsection{Interpretation}

Both systems reach perfect Recall@5 on this small fixture because the relevant event is usually somewhere in the top five results. The more meaningful difference is ranking quality: the hybrid system consistently places the relevant event first, while the keyword baseline often ranks the attendee record itself above the matching event.

That pattern matches the current implementation. The hybrid system benefits from concept expansion, structured intent scoring, and optional dense similarity, while the keyword baseline only sees surface token overlap.

\section{Limitations and Future Work}

The project still inherits strong limits from the source dataset:

\begin{itemize}
\item attendee roles are inferred heuristically rather than observed directly
\item asks/offers are synthetic placeholders derived from attendance context
\item quality depends on topic mappings in the ontology and on event naming quality
\item embedding support is optional and depends on local model/index availability
\end{itemize}

Promising next steps are:

\begin{enumerate}
\item add richer benchmark sets with manually judged relevance, not only weak labels
\item distinguish more event types and attendee intents beyond the current participant/session framing
\item expose score breakdowns and evaluation views directly in the UI
\item add safer profile enrichment pipelines only where external evidence is explicit
\end{enumerate}

\subsection{Local reproduction}

\textbf{1. Create a virtual environment and install dependencies}

\begin{verbatim}
python3 -m venv .venv
.venv/bin/python -m pip install -r requirements.txt
\end{verbatim}

\textbf{2. Import the Kaggle dataset}

\begin{verbatim}
python3 -m conference_matching.kaggle_import --input /path/to/event-attendance-dataset.zip
\end{verbatim}

Or, if using \texttt{kagglehub}:

\begin{verbatim}
.venv/bin/python -m pip install kagglehub
.venv/bin/python -m conference_matching.kaggle_import
\end{verbatim}

\textbf{3. Start the server}

\begin{verbatim}
.venv/bin/python server.py
\end{verbatim}

\textbf{4. Run evaluation}

\begin{verbatim}
.venv/bin/python -m conference_matching.evaluation
\end{verbatim}

\textbf{5. Run tests}

\begin{verbatim}
.venv/bin/python -m unittest discover -s tests
\end{verbatim}
