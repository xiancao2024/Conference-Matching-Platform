\section*{Abstract}

Large conferences make it difficult to turn informal networking goals into concrete introductions. This project implements a \textbf{retrieval-augmented} attendee matching system: a hybrid retriever ranks people from a \textbf{10k+ row} GTC-style profile table, then a \textbf{locally served} LLM (Ollama) generates short ``why connect'' text conditioned on \textbf{retrieved profile fields}, reducing unconstrained hallucination relative to pure chat.

\section{Course alignment (RAG)}

The course frames \textbf{Retrieval Augmented Generation (RAG)} as pairing an LLM with external data. Our system implements that pattern at the application layer:

\begin{itemize}
\item \textbf{External data / database:} normalized attendee records (\texttt{data/conference\_gtc.json}); dense vectors (\texttt{data/embeddings.npy}); FAISS index (\texttt{data/faiss.index}).
\item \textbf{Scale ($\geq$ 10k entries):} synthetic wide CSV from \texttt{scripts/generate\_gtc\_wide\_csv.py} (e.g.\ 10{,}000 rows) imported via \texttt{conference\_matching.gtc\_import}.
\item \textbf{Local LLMs:} Ollama on the same host or VM; no cloud LLM API in the default path.
\item \textbf{Front end:} browser UI (\texttt{static/}) with HTTP API (the assignment mentions Streamlit; we ship HTML/JS, which satisfies a user-facing interface).
\item \textbf{Verifiable grounding:} top matches expose structured fields (bio, interests, \texttt{source\_events}) passed into the LLM prompt; retrieval is deterministic without the LLM.
\end{itemize}

\noindent\textbf{Note on ``citations.''} Document-centric RAG often cites article passages. Our domain is structured profiles: verifiability comes from \textbf{grounding explanations in the same retrieved fields} shown in the UI. Future work can add row IDs or organizer portal links.

\section{Heilmeier Catechism}

\begin{enumerate}
\item \textbf{Goal:} Help attendees answer ``who should I meet?'' from natural language over a large roster.
\item \textbf{Status quo:} spreadsheets, ad-hoc search, or generic chatbots without a grounded attendee index.
\item \textbf{What's new:} hybrid retrieval plus local LLM explanations conditioned on retrieved profiles---not inventing attendees.
\item \textbf{Stakeholders:} participants and organizers who want actionable intros without fine-tuning on private data.
\item \textbf{Risks:} synthetic data is not real registration; CPU LLM latency; explanations must stay tied to retrieved text.
\end{enumerate}

\section{Motivation and impact}

Conference ROI is often limited by search friction. A future iteration with organizer-approved data could power on-site networking with auditable provenance.

\section{Background and related work}

Hybrid retrieval (sparse + dense + rules) is standard in search and RAG. Local inference (Ollama) satisfies on-metal or GCP deployment without third-party APIs.

\section{System architecture}

\textit{Data flow:} CSV $\rightarrow$ normalized JSON $\rightarrow$ optional MiniLM embeddings $\rightarrow$ FAISS index. \textit{Query flow:} browser $\rightarrow$ \texttt{POST /api/match} $\rightarrow$ hybrid ranker $\rightarrow$ top-$k$ profiles $\rightarrow$ optional Ollama \texttt{llm\_reason} from retrieved fields $\rightarrow$ UI.

\begin{verbatim}
[Browser] --POST /api/match--> [Hybrid matcher: lexical + concepts +
  structured + FAISS/MiniLM] --> [Top-k attendee records]
       --> [Ollama: explain using retrieved profile text only] --> [UI]
\end{verbatim}

\section{Modeling methodology and robustness}

\begin{itemize}
\item \textbf{Knowledge cutoff:} roster lives in external files, not model weights.
\item \textbf{Hallucinated people:} LLM only sees top-$k$ retrieved snippets.
\item \textbf{Score diversity:} multiple signals reduce over-reliance on a single embedding cosine.
\item \textbf{Separation of concerns:} retrieval is reproducible; generation is optional.
\end{itemize}

\section{Implementation}

\begin{itemize}
\item \texttt{conference\_matching/gtc\_import.py}: wide CSV $\rightarrow$ JSON
\item \texttt{conference\_matching/engine.py}: hybrid scoring, FAISS, payloads
\item \texttt{conference\_matching/llm.py}: Ollama explanations
\item \texttt{conference\_matching/server.py}: API + static UI
\item \texttt{Dockerfile}: containerized deployment
\end{itemize}

\section{Evaluation and analysis}

First-time embedding build on CPU dominates startup for 10k profiles; caching avoids re-encoding. LLM quality is constrained by prompting with retrieved fields.

\section{Limitations and future work}

Synthetic data; add row-level provenance links; optional GPU for Ollama.
