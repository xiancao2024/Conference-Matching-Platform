Abstract

This project implements a retrieval-augmented attendee matching system: a hybrid retriever ranks people from a 10k+ row GTC-style profile table, then a locally served LLM (Ollama) generates short “why connect” text conditioned on retrieved profile fields, reducing unconstrained hallucination relative to pure chat.

1 Course alignment (RAG)

The course defines RAG as pairing an LLM with external data. We retrieve from normalized JSON (conference_gtc.json), dense vectors (embeddings.npy), and a FAISS index (faiss.index). Scale uses synthetic 10k CSV generation + import. The LLM is local (Ollama). The front end is a browser UI (static/); the assignment mentions Streamlit—we ship HTML/JS. Verifiability: explanations use the same structured fields as retrieval (bio, interests, source_events), not free-form world knowledge.

2 Heilmeier Catechism (summary)

  1. Goal: “Who should I meet?” over a large GTC-style roster.
  2. Today: spreadsheets and generic chat without a grounded index.
  3. New: hybrid retrieval + local LLM explanations grounded on retrieved profiles.
  4. Stakeholders: attendees and organizers.
  5. Risks: synthetic data; CPU LLM latency; keep LLM tied to retrieved text.

3 System architecture (text diagram)

[Browser] --POST /api/match--> [Hybrid matcher: lexical + concepts + structured + FAISS/MiniLM]
       --> [Top-k attendee records + fields]
       --> [Ollama: llm_reason from retrieved profile text only] --> [UI]

4 Modeling methodology & robustness

  • External roster addresses LLM knowledge cutoff for attendee facts.
  • LLM does not invent people; it only sees top-k retrieved snippets.
  • Multiple retrieval signals reduce over-reliance on a single embedding score.
